# Communication Rules

이 문서는 COMMUNICATION_LOG.md에서 추출된 재발방지 규칙을 누적하는 문서다.
**모든 코딩 세션 시작 전에 AI에게 이 문서를 먼저 읽힌다.**

## 운영 원칙

- 세션 종료 후 로그를 남기고, 반복 가치가 있는 규칙만 이 문서에 추가한다.
- 규칙은 감상이 아니라 **실행 가능한 문장**으로 적는다.
- 각 규칙에는 근거 세션 번호를 붙인다.
- **졸업 기준:** 연속 5세션 동안 위반 없는 규칙은 Archive로 이동한다.
- **상한선:** Active Rules는 15개 이하로 유지한다. 넘으면 합치거나 졸업시킨다.

## Pre-Request Checklist

작업을 AI에게 넘기기 전에 아래 세 가지를 반드시 확인한다.

- [ ] **숨은 의도를 말했는가?** — 이 작업의 진짜 목적, 최적화 방향, 우선순위.
- [ ] **제약조건을 말했는가?** — 수정 범위, 건드리면 안 되는 영역, 기존 패턴.
- [ ] **완료 기준을 말했는가?** — 뭘 봐야 "됐다"인지, 엣지 케이스, 테스트 시나리오.

---

## Prompting Principles (내가 지킬 것)

- 작업의 **목적**을 첫 문장에 적는다.
- 기능 구현 / 버그 수정 / 리뷰 / 리팩터 중 어떤 작업인지 명시한다.
- 빠른 임시 수정인지, 운영 수준의 견고한 수정인지 밝힌다.
- 수정 가능 범위와 건드리면 안 되는 영역을 분리해서 적는다.
- 완료 기준을 가능하면 테스트 케이스로 적는다.

---

## AI Behavior Rules (AI에게 요구하는 것)

- 요구사항이 애매하면 구현 전에 숨은 목적과 우선순위를 **추정해서 드러낸다.**
- 제약조건이 비어 있으면 수정 범위를 **보수적으로** 잡고, 그 가정을 명시한다.
- 완료를 주장하기 전에 **확인한 것 / 확인 못 한 것**을 분리해서 말한다.
- 버그 수정에서는 재현 절차, 기대 동작, 수정 후 검증 결과를 함께 정리한다.
- 큰 가정을 할 때는 구현 전에 한 줄로 명시하고, 틀리면 교정받을 여지를 남긴다.
