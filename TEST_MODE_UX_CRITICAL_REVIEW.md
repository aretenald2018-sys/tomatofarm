# 테스트모드 UX 비판 리뷰 및 v3 개선안

작성일: 2026-05-02  
범위: 테스트모드 UX/서비스 시나리오 재설계 제안. 실제 앱 코드 구현 아님.  
목업: `mockups/test-mode-v3-direct-board.html`

---

## 1. 총평

현재 테스트모드는 기능은 늘었지만 사용자가 느끼는 주체성은 오히려 낮다. 사용자가 원했던 것은 "AI가 추천한 종목 묶음"이 아니라, 과거 엑셀에서 느꼈던 다음 감각이다.

1. 오늘 들어야 할 무게가 바로 보인다.
2. 그 무게가 6주 목표 안에서 어디쯤인지 보인다.
3. 마음에 안 들면 즉시 바꿀 수 있다.
4. 바꾼 값이 다음 계획에 반영된다.
5. 전체 성장판을 보면 내가 앞으로 성장하고 있다는 느낌이 든다.

현재 UX는 이 핵심을 주변 정보가 가리고 있다. 테스트모드는 "추천 시스템"처럼 보이고, 사용자는 "내 계획을 운영하는 사람"이 아니라 "시스템이 만든 카드를 승인하는 사람"처럼 느낀다.

---

## 2. 최고 UX 디자이너 관점의 비판

### 2.1 정보 구조가 사용자의 실제 질문 순서와 다르다

운동 직전 사용자의 질문은 다음 순서다.

1. 오늘 뭐 하지?
2. 몇 kg으로 하지?
3. 이게 내 6주 계획에서 맞는 무게인가?
4. 컨디션에 따라 낮추거나 올릴 수 있나?
5. 끝나고 실제 수행값을 어떻게 남기나?

하지만 현재 화면은 `청사진`, `설정`, `보강 추천`, `근거`, `기구`, `정산`이 병렬로 떠서 사용자가 먼저 해석해야 한다. 토스나 애플식으로 보면 첫 화면의 과업 정의가 흐리다.

### 2.2 트랙이 설명은 되지만 조작되지 않는다

볼륨트랙/중량트랙은 현재 상태 배지처럼 보인다. 사용자가 "오늘은 강도 말고 볼륨으로 가야겠다"라고 판단해도 즉시 바꿀 수 있다는 느낌이 약하다.

트랙은 추천 결과가 아니라 오늘 세션의 핵심 입력값이다. 따라서 카드 내부 라벨이 아니라 화면 상단의 큰 segmented control이어야 한다.

### 2.3 무게가 추천값이 아니라 고정값처럼 보인다

현재 카드는 "77.5kg x 12"를 보여주지만, 이 값이 조정 가능한 계획값인지, AI가 잠근 값인지, 지난 기록에서 계산된 값인지가 즉시 드러나지 않는다.

운동 앱에서 중량은 사용자가 가장 민감하게 통제하고 싶어 하는 숫자다. 따라서 무게는 텍스트가 아니라 stepper가 되어야 한다.

### 2.4 CTA가 너무 많고 위계가 없다

`시작`, `설정`, `수락`, `수정`, `거절`, `공통/헬스장별 기구`, `정산`이 비슷한 무게감으로 노출된다. 사용자는 "이 화면에서 지금 눌러야 하는 것"을 잃는다.

첫 화면 CTA는 1개여야 한다.

- 주 CTA: `오늘 운동 시작`
- 보조 액션: `계획 조정`
- 숨겨도 되는 액션: 기구 관리, 정산, 프레임워크 변경

### 2.5 색의 의미가 일관되지 않다

빨강은 브랜드, 경고, 진행, 트랙, CTA를 모두 담당하고 있다. 노란색, 초록색, 부위색도 섞인다. 색이 많아질수록 사용자는 색을 의미로 읽지 않고 장식으로 읽는다.

v3에서는 색을 4개 역할로 제한한다.

| 역할 | 색 |
|---|---|
| 브랜드/주 CTA | Tomato red |
| 계획/예측 | Neutral blue-gray |
| 실제 완료 | Green |
| 주의/정체 | Amber |

부위색은 첫 화면에서 제거한다. 부위 구분은 텍스트와 순서로 충분하다.

### 2.6 "성장판"이 화면의 주인공이 아니다

성장판이 핵심인데 현재는 카드 중 하나로 보인다. 엑셀의 강점은 한 화면에서 시간축과 종목축을 동시에 보는 것이었다. 모바일에서는 전체 엑셀표를 그대로 복제하면 안 되지만, 첫 화면에 최소한 "오늘 행"과 "6주 목표 행"은 같은 문맥 안에 있어야 한다.

### 2.7 설정과 실행이 섞여 있다

운동 직전 화면에서 framework, deload, target set 같은 설정값이 보이면 사용자의 실행 흐름이 끊긴다. 설정은 온보딩과 별도 편집 시트에 있어야 한다. 오늘 화면에는 오늘 바꿀 수 있는 것만 남겨야 한다.

## 3. v3 North Star

> 테스트모드는 추천 카드 모음이 아니라, 사용자가 오늘의 계획 무게를 직접 조정하고 수행하며 6주 성장판을 운영하는 화면이다.

핵심 UX 문장:

> "오늘 벤치 77.5kg. 필요하면 75나 80으로 바꾸고 바로 시작한다. 바꾸면 6주 계획도 같이 조정된다."

## 4. v3 서비스 시나리오

### 4.1 화면 수를 줄인다

기존 v2는 화면이 많다. v3는 3개 화면으로 줄인다.

| 화면 | 역할 | 진입 |
|---|---|---|
| Today Board | 오늘 수행/조정 | 테스트모드 첫 화면 |
| Cycle Board | 6주 전체표 | Today Board의 `전체표` |
| Plan Editor | 벤치마크/기구/증량폭 편집 | Today Board의 `계획 조정` |

`정산`, `기구 관리`, `프레임워크`는 Plan Editor 안으로 들어간다.

### 4.2 첫 화면의 구조

1. 상단: Week 3/6, 오늘 트랙 전환
2. 메인: 오늘의 벤치마크 3-5개
3. 각 종목: 계획 무게 stepper, 반복수, 실제 입력
4. 하단 sticky CTA: 오늘 운동 시작
5. 보조: 전체표, 계획 조정

### 4.3 트랙 전환

볼륨/강도 트랙은 자동 추천되지만 사용자가 즉시 바꿀 수 있어야 한다.

- 기본값: 사이클 규칙에 따라 자동
- 사용자가 바꾸면: 오늘만 변경
- 바꾼 이유 선택은 선택사항
- 다음 추천에는 "사용자가 최근 2회 강도 트랙을 피함" 정도만 반영

### 4.4 무게 조정

무게는 카드 안에서 바로 조정한다.

- `-2.5`
- 현재 kg input
- `+2.5`
- `지난 성공값`
- `계획값으로 복구`

무게를 바꾸면 카드 하단 문장이 즉시 바뀐다.

예:

- 계획보다 -2.5kg. 오늘만 낮추고 다음 주는 원래 계획 유지.
- 계획보다 +2.5kg. 성공하면 다음 주 목표를 앞당김.

### 4.5 보강 추천의 위치

보강 추천은 메인 카드가 아니다. 오늘의 벤치마크를 수행한 뒤 부족한 세트가 있을 때만 "마지막 10분" 섹션으로 나온다.

이렇게 바꾼다.

- 기존: 추천 카드 여러 장 → 사용자가 선택
- v3: 오늘 핵심 3-5종 수행 → 부족분 자동 1-2개만 제안

### 4.6 색/컴포넌트 원칙

- 카드 radius 8px.
- 한 화면에서 primary CTA는 하나.
- 빨강은 CTA와 현재 진행 위치에만 사용.
- 트랙은 색이 아니라 segmented control의 선택 상태로 표현.
- 부위별 색상칩 제거.
- 긴 설명은 접고, 기본 화면에는 수치만 둔다.
- 모든 중량/횟수는 같은 mono number 스타일.

## 5. v3 목업 구성

목업 파일: `mockups/test-mode-v3-direct-board.html`

## 6. 구현 방향

v2 구현을 완전히 버릴 필요는 없다. 다만 화면의 우선순위를 바꿔야 한다.

### 유지

- `max_cycle` 데이터 모델
- 6주 progression 계산
- equipment pool 모델
- recommendationMeta에 cycle/track 저장

### 변경

- `renderMaxCycleDashboard()`를 테스트모드 첫 화면의 중심으로 승격
- `setupHtml`은 접힌 `계획 조정` 시트로 이동
- 추천 카드는 `마지막 10분 보강` 섹션으로 강등
- `applyMaxSuggestion`보다 `applyTodayPlan`이 먼저 와야 함
- 카드의 무게는 텍스트가 아니라 stepper/input으로 변경

### 제거 또는 숨김

- 첫 화면의 framework copy
- 첫 화면의 기구 관리 CTA
- 첫 화면의 정산 CTA
- 빨강/노랑/초록/부위색이 섞인 카드 테두리

---

## 7. 성공 기준

사용자가 테스트모드 첫 화면에서 5초 안에 다음 질문에 답할 수 있어야 한다.

1. 오늘 몇 주차인가?
2. 오늘은 볼륨인가 강도인가?
3. 오늘 벤치/랫풀다운/스쿼트를 몇 kg 해야 하나?
4. 그 무게를 바꿀 수 있나?
5. 바꾸면 6주 계획에 어떤 영향이 있나?

이 5개가 한 화면에서 해결되지 않으면 테스트모드는 아직 실패다.
